## 1. Резюме

Onsite Backend — REST API на Node.js (Fastify 5 + PostgreSQL 16), превращающее существующее offline-first мобильное mini-CRM Onsite в мультипользовательскую систему: диспетчер создаёт и назначает заявки, техники исполняют их в поле и синхронизируют статусы и фотоотчёты при появлении сети. Ядро проекта — протокол офлайн-синхронизации с идемпотентными мутациями, курсором изменений и server-authoritative разрешением конфликтов. Проект выполняется одним разработчиком как портфолио-работа; ключевые метрики успеха — 0 потерянных мутаций в e2e-сценарии «офлайн-смена», p95 API ≤ 200 мс и покрытие доменной логики тестами ≥ 80 %. Сроки относительные (даты в спецификации не проставлены намеренно): три вехи — M0 (скелет), MVP (синк-протокол), v1.0 (push и наблюдаемость), суммарно ~13–14 недель при full-time **[предположение]**.

## 4. Не входит в проект (Out of Scope)

Перенесено из §2.2 спецификации:

- Веб-панель диспетчера (UI) — обоснование: в MVP диспетчер работает через OpenAPI UI; панель — отдельный будущий проект.
- Доработка мобильного клиента под синхронизацию — обоснование: отдельная спецификация и фаза после бэкенда (§9.4).
- Отчётность и аналитика (агрегаты по техникам, экспорт) — обоснование: кандидат на v1.1, не влияет на ядро синк-протокола.
- Биллинг, мультитенантность, чат, геотрекинг маршрутов — обоснование: портфолио-масштаб, одна организация (допущения §8).
- Самостоятельная регистрация пользователей — обоснование: аккаунты создаёт диспетчер или сид (§2.2, допущения §8); self-service восстановление пароля также вне scope (решение §9.8).
- Retention-policy фотоотчётов — обоснование: в MVP/v1.0 хранение бессрочно, авто-удаление — кандидат v1.1 (решение §9.3).

## 5. Технический стек и архитектура

Стек зафиксирован в спецификации (§5.3), заказчик ограничил только runtime (Node.js):

| Слой             | Технология                        | Обоснование                                                                  |
| ---------------- | --------------------------------- | ---------------------------------------------------------------------------- |
| Runtime          | Node.js 24 LTS, TypeScript strict | Ограничение проекта; единый язык и конвенции с мобильным клиентом            |
| HTTP API         | Fastify 5 + TypeBox               | Schema-first валидация с генерацией OpenAPI, производительность, минимум магии |
| БД               | PostgreSQL 16                     | Транзакционность синк-батчей, `bigserial` для курсора, JSONB для payload'ов  |
| ORM и миграции   | Drizzle ORM + drizzle-kit         | SQL-близкий, строгая типизация, не прячет транзакции (критично для FR-09)    |
| Хранилище файлов | MinIO / S3 API v4                 | Presigned URL снимают трафик фото с API-процесса                             |
| Push             | expo-server-sdk (outbox-worker)   | Клиент уже на Expo, токены доступны без нативной работы с FCM/APNs           |
| Инфраструктура   | Docker Compose, GitHub Actions    | Стек одной командой; CI-конвенции заданы мобильным репо                      |

Архитектура (компоненты, модель данных, API-контракт) — см. спецификацию, §5.1–5.6.

## 7. Ресурсы

### Команда (соло-режим)

Проект выполняет один разработчик (Dmitriy) — весь стек от схемы БД до деплоя. Доступность в часах в неделю в спецификации не зафиксирована; календарные оценки в §8 построены из расчёта full-time ~40 ч/нед **[предположение]** — при доступности 20 ч/нед календарный срок удваивается. ## 9. Критерии приёмки (Definition of Done)

- [ ] e2e «офлайн-смена» (50+ мутаций, 3 обрыва сети) — 0 потерянных и 0 задублированных мутаций; тест в CI (G-01, NFR-04)
- [ ] Повторная отправка того же батча мутаций — все `duplicate`, состояние БД не меняется; тест идемпотентности в CI (FR-09)
- [ ] Матрица переходов статусов покрыта юнит-тестами полностью; недопустимый переход — 409 `invalid_transition` (FR-07)
- [ ] Сценарий конфликта US-06 (техник офлайн ставит Done, диспетчер отменяет) покрыт интеграционным тестом: `conflict` + снимок Cancelled (FR-10)
- [ ] p95 ≤ 200 мс, p99 ≤ 500 мс при 50 RPS — подтверждено нагрузочным прогоном (NFR-01)
- [ ] Покрытие доменной логики (переходы, синк-протокол) тестами ≥ 80 % — отчёт coverage в CI (NFR-12)
- [ ] OpenAPI-спека валидна (openapi-валидатор в CI); `/docs` доступен без чтения серверного кода (FR-17, US-07)
- [ ] `docker compose up` поднимает весь стек одной командой; повторный запуск сида не создаёт дубликатов (FR-16, US-07)
- [ ] Push о назначении доставляется на все активные устройства техника ≤ 30 с (FR-13, FR-14, G-02)
- [ ] OWASP API Top-10 чек пройден перед v1.0; секреты вне git, пароли argon2id, JWT RS256 (NFR-05, NFR-06)
- [ ] Все P0-задачи закрыты и приняты
